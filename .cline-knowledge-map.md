## 1. What is BetterBundle?

A **Shopify App** that provides **AI-powered product bundle recommendations** and **commission-based billing** for Shopify merchants. It's a monorepo with two main services:

| Service          | Language/Framework      | Purpose                                                                     |
| ---------------- | ----------------------- | --------------------------------------------------------------------------- |
| `better-bundle/` | Remix (React) + Node.js | Shopify app frontend + API. Merchant dashboard, billing UI, onboarding.     |
| `python-worker/` | Python (FastAPI)        | ML recommendations engine, Kafka consumers, data processing, billing logic. |
| `admin/`         | Django (Python)         | Internal admin panel for managing shops, billing, revenue data.             |

## 3. Core Flows (How Things Work)

### A. Billing Flow (The most complex flow)

This is a **commission-based** billing model (NOT flat subscription pricing).

1. **Trial Phase**: Each shop starts with a trial. They earn `commission_rate` % on revenue until they hit `trial_threshold` (default $75).
2. **Transition to Paid**: Once trial revenue >= threshold, merchant must create a Shopify subscription with a usage cap.
3. **Usage Charging**: Commission is calculated on every order attributed to bundles. Commission records go through states: `PENDING` → `RECORDED` (on Shopify) → `NOT_APPLICABLE` if capped.
4. **Billing Cycles**: Monthly cycles track usage against cap. Cycles have statuses: DRAFT → ACTIVE → COMPLETED → CANCELLED

**Key Files for Billing:**

- `better-bundle/app/features/billing/services/billing.service.ts` → Main billing state machine (trial / active / pending / completed)
- `better-bundle/app/features/billing/components/*.tsx` → UI components for each state
- `better-bundle/app/features/billing/types/billing.types.ts` → Type definitions
- `better-bundle/app/middleware/serviceSuspension.ts` → Suspends service if billing fails
- `better-bundle/app/routes/api.billing.*.tsx` → Billing API endpoints
- `python-worker/app/consumers/kafka/billing_consumer.py` → Processes billing events
- `python-worker/app/domains/billing/` → Billing domain logic
- `python-worker/app/repository/CommissionRepository.py` → Commission data access
- `python-worker/app/repository/BillingCycleRepository.py` → Billing cycle data access

### B. Recommendation Flow

1. Data is collected from Shopify (orders, products, collections, interactions) via API and Kafka events.
2. Feature engineering computes product embeddings, collection features, and customer profiles.
3. ML models (FP-Growth, hybrid collaborative filtering) generate recommendations.
4. Recommendations are stored in PostgreSQL and served via REST API to the Remix app.

**Key Files for Recommendations:**

- `python-worker/app/recommandations/` → All recommendation engines
  - `hybrid.py` → Hybrid collaborative + content-based filtering
  - `fp_growth_engine.py` → Frequent pattern mining for bundle discovery
  - `frequently_bought_together.py` → FBT recommendations
  - `recommendation_executor.py` → Orchestrates all recommendation strategies
  - `product_embeddings.py` → Product vector embeddings
  - `smart_selection_service.py` → Smart bundle selection logic
  - `business_rules_filter.py` → Filters recommendations by business rules
- `python-worker/app/api/v1/recommendations.py` → Recommendation API endpoint
- `python-worker/app/consumers/kafka/data_collection_consumer.py` → Collects data from Kafka
- `python-worker/app/consumers/kafka/purchase_attribution_consumer.py` → Processes purchase attribution
- `python-worker/app/repository/` → Data access layer for recommendations

### C. Purchase Attribution Flow

1. Orders come in via Shopify webhook → Kafka event.
2. Purchase attribution consumer determines which products were part of a bundle and calculates attributed revenue.
3. Commission is calculated based on attributed revenue × commission rate.
4. Commission records are created and later synced to Shopify via usage records.

**Key Files:**

- `python-worker/app/consumers/kafka/purchase_attribution_consumer.py` → Attribution logic
- `python-worker/app/consumers/kafka/shopify_usage_consumer.py` → Shopify usage record sync
- `python-worker/app/repository/PurchaseAttributionRepository.py` → Data access

### D. Data Flow Architecture

```
Shopify Merchant ←→ Remix App (better-bundle) ←→ PostgreSQL
                        ↕ (REST API)
                  Python Worker (FastAPI)
                        ↕
                  Kafka ←→ Gorse (ML Engine)
                        ↕
                Data Collection Consumers
```

**Logging/Monitoring Stack:**

- Loki (log aggregation) + Promtail (log collector) + Grafana (visualization)
- Grafana at http://localhost:3001 (dev), Loki at port 3100

## 4. Database Schema Summary

All tables are defined in `better-bundle/prisma/schema.prisma` (1127 lines).

| Table                   | Purpose                       | Key Fields                                                           |
| ----------------------- | ----------------------------- | -------------------------------------------------------------------- |
| `shops`                 | Shop/merchant records         | id, domain, status, currency_code                                    |
| `shop_subscriptions`    | Subscription plans per shop   | shop_id, pricing_tier_id, trial_revenue, status                      |
| `billing_cycles`        | Monthly billing cycles        | shop_subscription_id, cycle_number, cap_amount, usage_amount, status |
| `commission_records`    | Individual commission charges | shop_id, order_id, commission_charged, status, billing_cycle_id      |
| `pricing_tiers`         | Pricing configurations        | trial_threshold_amount, commission_rate, currency                    |
| `purchase_attributions` | Links purchases to bundles    | order_id, bundle_id, attributed_revenue                              |
| `products`              | Synced product catalog        | shop_id, title, price, tags                                          |
| `product_embeddings`    | Vector embeddings for ML      | product_id, embedding vector                                         |
| `collection_data`       | Collection/group data         | shop_id, title, product_count                                        |
| `collection_features`   | ML features for collections   | shop_id, engagement_score, conversion_rate                           |
| `user_interactions`     | Customer interactions         | shop_id, user_id, product_id, interaction_type                       |
| `user_sessions`         | User session tracking         | shop_id, user_id, session_data                                       |
| `orders`                | Synced order data             | shop_id, order_id, total_price, customer_id                          |
| `billing_invoices`      | Generated invoices            | shop_subscription_id, total_amount, status                           |

Full schema at: `better-bundle/prisma/schema.prisma`

## 5. Important Architecture Decisions & Gotchas

### ✅ Critical Rules:

- **Always check Shopify GraphQL as source of truth** for subscription status, not just DB (see `getShopifySubscriptionStatus`)
- **Commission state machine**: `PENDING → RECORDED → NOT_APPLICABLE`. REJECTED means the charge exceeded cap.
- **Trial revenue** is tracked on the subscription record (`trial_revenue` field), not a separate table.
- **Billing cycles** use a unique constraint: one active cycle per subscription.
- **Gorse is the ML backend** - all recommendations go through Gorse's REST API.
- **Redis is used** for caching recommendations and session data.
- **Kafka topics** coordinate between the Remix app and Python worker for async processing.

### ❌ Common Pitfalls to Avoid:

- Don't rely solely on DB subscription status — always verify with Shopify GraphQL.
- Don't forget to check `billing_phase` on commission records (TRIAL vs PAID).
- The recommendation engine expects pre-computed embeddings. Don't assume they exist.
- The `commission_overflow` field captures charges that exceed the cap.
- `shopify_usage_record_id` must be unique per commission record (Shopify constraint).

### 🔍 Debugging Tips:

- **Billing issues**: Check `commission_records` table → look at `status` and `billing_phase`
- **Recommendation issues**: Check Gorse logs → `docker logs betterbundle-gorse-dev`
- **Kafka issues**: Use Kafka UI at http://localhost:8080 (dev)
- **Payment suspense**: Check `serviceSuspension.ts` middleware
- **General logs**: Grafana at http://localhost:3001 (user: admin, pass: admin)

## 6. How to Answer Common Questions

> **"How does billing work?"** → See Section 3A. Key files: `billing.service.ts`, billing Kafka consumers, commission repository.

> **"How do recommendations work?"** → See Section 3B. Key files: `recommandations/` directory, data collection consumers.

> **"Where are database tables defined?"** → `better-bundle/prisma/schema.prisma`

> **"How do I debug a billing issue?"** → Check `commission_records.status`, then `billing_cycles`, then check Shopify via GraphQL. See Debugging Tips.

> **"What architecture pattern is used?"** → Event-driven microservices with Kafka. Remix app calls Python worker via REST. Python worker processes async events via Kafka consumers.

> **"How is the trial system set up?"** → `BillingService.getBillingState()` checks `trial_revenue < trial_threshold`. If true → trial_active. If false → needs subscription.

> **"Where is the Gorse ML configuration?"** → `config.toml` in the root directory, mounted into the Gorse Docker container.
